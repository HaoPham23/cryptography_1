  • Ex: Real world examples
    • Project Venona (1941-1946):

      In the 1940s, Russians used one time pad to encrypt various messages. They generated key pads by throwing dice. Because of the laziness, they only used one key pad to encrypt multiple messages. And US intelligence was able to intercept these multiple time pads. The result is that over a several years, they decrypted something like 3,000 plain texts just by this attack.

    • MS-PPTP (windows NT):

      PPTP: Point-to-Point Transfer Protocol. The communication between the client and server used the same key k for encrypt m (messages) and s (response).

    • 802.11b WEP:

      The client and the access point both share a same key k. When they send a message m, m will be concatenate with a CRC(m) function (not important!), and k will be concatenate with a 24-bit value IV. The IV then increases by 1 for the next message. The problem is that IV is just 24-bit, so after 2242^{24}224﻿ messages, or even after a power cycle, IV will be repeated!

Attack 2: no integrity (OTP is malleable)

Modifications to CT are undetected and have predictable impact on plaintext. Take an example: Suppose the user sends an email that starts with the word “From: Bob”. The attacker intercepts the corresponding ciphertext. He doesn’t know what the plain text is, but he knows it is from Bob. He wants to make it look like this message is from “Eve”. He can XOR with a certain three characters “Bob”. He CAN do this, without any knowledge about the key or the remain content in plaintext.

  • More formally

    Suppose he has CT ccc﻿ and wants to fake the message mmm﻿ to fff﻿. All he need to do is replace ccc﻿ by c⊕(m⊕f)c\oplus (m\oplus f)c⊕(m⊕f)﻿. When the receiver decrypts this CT, instead of D(c,k)=mD(c,k)=mD(c,k)=m﻿, what really happens is D(c⊕(m⊕f),k)=(m⊕k)⊕(m⊕f)⊕k=fD(c\oplus (m\oplus f), k)=(m\oplus k)\oplus (m \oplus f) \oplus k = fD(c⊕(m⊕f),k)=(m⊕k)⊕(m⊕f)⊕k=f﻿. Old example: CSS (1996) (hardware)

CSS: Content Scramble System

It is badly broken.

Used to encrypt DVD movies. Based on a hardware called Linear feedback shift register (LFSR).

  • What is LFSR?

    It’s a register that has a certain special cells (called taps). Every clock cycle, it XOR these taps, shift right 1 bit, and add the XOR result to the leftmost bit. The leftmost bit is used for PRG.

  • How does it work?

    There are two LFSRs, 17-bit and 25-bit.

    1+2x8 first seed bits →\rightarrow →﻿17-bit LFSR

    1+3x8 last seed bits →\rightarrow→﻿ 25-bit LFSR

    For every round, these LFSR produce 8 bits, then add together (mod 256) to create 1 byte key. This key is used to encrypt with the appropriate byte of the movie.

  • How to break it?

    In fact, since the DVD encryption is using MPEG files, the attackers can know a prefix of the plaintext. (suppose 20 bytes). For every value of 2172^{17}217﻿ of 17-bit LFSR, they run the first LFSR for 20 bytes. Then substract the prefix from this 20 bytes, check if the 20 bytes result came from a 25-bit LFSR or not!

Semantic Security

Attacker’s abilities: obtains one CT

Security requirement: CT should reveal no “info” about PT

A semantically secure cryptosystem: is one where only negligible information about the PT can be revealed from the CT.
