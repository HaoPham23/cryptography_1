Padding Oracle Attack

During decryption:

  • If pad is invalid, send decryption_failed alert
  • If mac is invalid, send bad_record_mac alert.

⇒\Rightarrow⇒﻿ Attacker learns info. about PT

⇒\Rightarrow⇒﻿ When decryption fails, do not explain why!

  • Note: If the decryptor doesn’t explain why the CT is rejected, attacker can oracle via timing:

Padding Oracle: Attacker can submit CT and learn if last bytes of PT are a valid pad.

Attack:

Attacker repeat with g=0,1,…,255 to learn last byte of m[1].

Then use 0x0202 pad to learn the next byte and so on: ⊕(g∣∣last-byte)⊕0x0202\oplus (g||\text{last-byte}) \oplus \text{0x0202}⊕(g∣∣last-byte)⊕0x0202﻿

Lesson:

  • Encryption-then-MAC would avoid this problem: MAC is checked first and CT discarded if invalid
  • MAC-then-CBC provides AE, but padding oracle destroys it

Attack on enc. length field

Let’s see SSH Binary Packet Protocol:

Danger: non-atomic decrypt, it use decrypted packet length before it is authenticated!!!

Attack: Attacker has one CT block c=AES(k,m) and it wants m. First, it sends c to server. The server will decrypt the 32 MSB bits of c as packet_length. (assume packet_length=5).

Then attacker sends byte-by-byte (trash bytes) to the server and count. When he sent 5 bytes, the server sends “MAC error”, the attacker will know that 32 MSB bits of m is equal to 5!.
